# Plan 02.03 — Migrate pages from direct seed imports → repos

## Goal

Replace every `import { <entity> } from "@/data/seed/<entity>"` in `src/app/` with `import { <entity>Repo } from "@/lib/repos"` + call `<entity>Repo.list()` / `<entity>Repo.get(id)`. Pull entity types from `@/lib/types` (no page-local `Seed*` aliases).

## Why

- DATA-01 #1 enforcement: pages must render from the shared source. Direct seed imports break the abstraction.
- DATA-02 enforcement: the wiring file becomes the only place that decides in-memory vs HTTP; pages don't know.

## Migration rules

1. **Replace import line.** `import { X } from "@/data/seed/<entity>"` → `import { xRepo } from "@/lib/repos"`.
2. **Replace type imports** for entity types → `import type { ... } from "@/lib/types"`.
3. **Drop `Seed*` aliases.** `type Invoice as SeedInvoice` → `type Invoice`; references in the file unalias.
4. **Replace data references.** `seedEquipment` / `equipment` (the array) → `equipmentRepo.list()`. For detail pages, `equipment.find((e) => e.id === id)` → `equipmentRepo.get(id)`. Repo's `get()` returns `undefined` on miss — caller's existing `if (!record)` branch keeps working.
5. **Keep `@/data/seed/_shared/*` imports** intact per D-06 (customers, depots, surveyors, etc.).
6. **Keep page-local row-type aliases** (`type CleaningRow`, etc.) — those are presentation reshapings (D-04).

## File-by-file scope

See 02-PATTERNS.md "Page migration table" — 30 files total. Mechanical regex/edit per the rules above.

## Per-page verification

For each migrated page:
- `import { ... } from "@/data/seed/<entity>"` line gone (except `_shared/*`)
- One of: `import { <entity>Repo } from "@/lib/repos"` OR `import { <entity1>Repo, <entity2>Repo } from "@/lib/repos"`
- Page logic unchanged otherwise
