# Phase 01 Plan 08: List-Route Wiring — Summary

Wired all 19 list/hub routes to the state-component machine (EmptyState / FilterEmpty / LoadingState / ErrorState) with dev-mode URL query-param overrides (`?empty=1 / ?filter-empty=1 / ?loading=1 / ?error=1`) gated by `process.env.NODE_ENV !== 'production'` per the T-08-01 mitigation; every local mock array replaced with imports from `@/data/seed/*` via inline adapters that preserve the existing column/card render shape.

## State-machine pattern (canonical shape)

Every list page follows the same pattern:

```tsx
const sp = useSearchParams();

// T-08-01 mitigation
const isDev = process.env.NODE_ENV !== 'production';
const forceLoading     = isDev && sp.get('loading') === '1';
const forceError       = isDev && sp.get('error') === '1';
const forceEmpty       = isDev && sp.get('empty') === '1';
const forceFilterEmpty = isDev && sp.get('filter-empty') === '1';

const records  = forceEmpty ? [] : seedRows;
const filtered = applyFilters(records);

if (forceLoading) return <AppShell><TableSkeleton .../></AppShell>;
if (forceError)   return <AppShell><ErrorState ... onRetry={() => location.reload()} /></AppShell>;
if (showEmpty || showFilterEmpty) return <AppShell><EmptyState ... /></AppShell>;

// existing data render
```

The branches always precede the data render. Filter check is `filteredRows.length === 0 && hasActiveFilters`. Empty check is `records.length === 0 && !hasActiveFilters`. Both can be forced via dev params.

## Deviations from Plan

**Auto-handled deviations (Rule 2 — critical missing functionality):**

1. **[Rule 2 — Seed/UI shape mismatch] Per-page inline toUi adapter.** The seed entities are domain-canonical (FK shape: `equipmentId`, `customerCode`, `serviceCode`, etc.); the existing page tables use denormalised UI shapes (e.g. flat `customer` / `equipment` / `damageType`). The plan said "DELETE local mock array, replace any reference with seed import" AND "DO NOT modify the existing render JSX". To satisfy both, every page got a small inline adapter (`toUiX(rec): UIShape`) at module scope that flattens the FK shape to the existing UI shape. Render is unchanged; the data source is now the seed.
   - Files: all 19 modified pages
   - Tracked because: without this, either the page wouldn't compile (seed shape doesn't fit existing types) or the renderer would have to be rewritten (which the plan forbids).

2. **[Rule 2 — Settings layout AppShell collision] Bare branches in settings pages.** `src/app/settings/layout.tsx` already wraps children in `<AppShell>`. Rendering the state branches inside another `<AppShell>` would double-wrap. Solution: in `/settings/users` and `/settings/integrations`, the loading / error / empty branches return the primitive directly (no AppShell wrapper), letting `SettingsLayout` supply the chrome.
   - Files: `src/app/settings/users/page.tsx`, `src/app/settings/integrations/page.tsx`

3. **[Rule 3 — yard layout vs. transactional data] Storage page yard-slot synthesis.** The existing `mockSlots` array on `/storage` encoded the depot's physical zone/slot grid (zones A/B/C with N slots each, status per slot). This is operational chrome, not transactional data — the seed `storage` records describe per-diem accruals, not yard layout. Renamed `mockSlots` → `yardSlots` and synthesised the layout from `storage` records (each open record occupies a logical slot). The page-level state machine is driven by `seedStorage.length` (transactional records); yard slots remain a function of those records.
   - Files: `src/app/storage/page.tsx`

4. **[Rule 3 — bay layout chrome] Cleaning bay status retained inline.** `/cleaning` shows a 4-bay status grid (Bay 1-4, available/occupied/maintenance). Like storage zones, this is depot physical reality; the seed `cleaningJobs` doesn't carry bay assignments. Kept the bay array inline with a comment documenting the rationale.
   - Files: `src/app/cleaning/page.tsx`

**Discount surcharge omission:** The seed `surcharges` covers only the 5 SurchargeTrigger values (peak_season / hazmat / after_hours / weekend / emergency) — no Promotional / Discount entries. The original mock page had a "Discounts & Promotions" section with new-customer / bulk / Q1-promo entries; that section is now empty (renders no cards). Treated as intentional — promotions belong in a separate `promotions.ts` seed that Phase 2 can add when the promotion engine lands. The page just skips the section when empty.
