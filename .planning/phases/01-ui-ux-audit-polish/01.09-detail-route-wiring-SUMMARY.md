# Phase 1 Plan 09: Detail-Route Wiring Summary

Wired all 11 detail routes (UI-SPEC §12.2 — 10 entity-detail + 1 edit variant) to the state-component machine. Every detail route now renders DetailSpinner on `?loading=1`, ErrorState on `?error=1`, and a gecko-styled NotFound with Levenshtein "Did you mean X?" suggestion when the URL `id` doesn't resolve in seed data.

## What Was Built

Three-branch render contract added to every detail page above the existing detail JSX:

```tsx
const isDev = process.env.NODE_ENV !== "production";          // T-09-01
const forceLoading = isDev && sp.get("loading") === "1";
const forceError   = isDev && sp.get("error") === "1";
const record = <entity>.find(r => r.<keyField> === id);

if (forceLoading) return <AppShell><DetailSpinner label={getLoadingLabel(ROUTE)} /></AppShell>;
if (forceError)   return <AppShell><ErrorState .../></AppShell>;
if (!record) {
  const suggestion = nearestReference(id, <entity>.map(r => r.<keyField>));
  // ...NotFound with Did-you-mean Link to /<route>/${encodeURIComponent(suggestion)}  // T-09-02
}

// existing detail render — uses `record` from seed lookup (no local mock for the entity)
```

Below the branches, every page's existing detail JSX is preserved. Local mock objects that previously held the entity's core fields were replaced with seed lookups. Decorative "mockChrome" objects (photos, contact info, history timelines, document lists, financial KPIs that the seed doesn't yet model) remain as visual chrome so the existing UI shape stays intact.

## Levenshtein Integration

Each NotFound branch calls `nearestReference(id, candidates)` from `src/lib/levenshtein.ts`. The helper:
- Returns the best candidate with Levenshtein distance ≤ 2 (case-insensitive), OR
- Falls back to a candidate that shares a ≥4-digit numeric prefix with the input (per D-03)
- Returns `null` if no candidate qualifies (no suggestion line rendered)

Example: visiting `/repair/REP-2026-XXXX` with seed reference `REP-2026-0001` triggers the numeric-prefix fallback (`2026` matches) → "Did you mean REP-2026-0001?" appears.

## Security — T-09 Threat Register Mitigations

**T-09-01 (Spoofing — dev-param phishing-via-URL):** Every `sp.get('loading')` / `sp.get('error')` check is preceded by `const isDev = process.env.NODE_ENV !== 'production'`. In production, `forceLoading` and `forceError` collapse to `false` regardless of URL contents, so an attacker cannot trick a user into seeing a fake loading or error state on a production deploy.

**T-09-02 (Tampering — XSS via `[id]` substitution):** All URL substitutions use React JSX children (not innerHTML), and every constructed href passes the user-controlled `id` through `encodeURIComponent`. Grep-verified in all 11 modified files. Special characters in the URL param (`<`, `>`, `&`, `"`, `'`, spaces) are percent-encoded before they reach the browser's URL bar.

**T-09-03 (Open redirect):** Accepted (per plan threat register). The `nearestReference` suggestion is computed from in-memory seed arrays, so the Link href can only point to a route prefix + a value drawn from a hardcoded list. No external URLs are reachable from this surface.

## Tariff Customer-Rates Dual-Purpose NotFound

The route `/tariff/customer-rates/[customerId]` is unusual: the URL param is a CUSTOMER code (e.g. `C-MSKU`), not a rate-override record id. The page distinguishes:

1. **Customer doesn't exist** (`!customer`) → true NotFound with Levenshtein suggestion against `customers.map(c => c.code)`.
2. **Customer exists but has zero rate overrides** (`ratesForCustomer.length === 0`) → plan-04's `not-found` variant copy ("This customer has no rate overrides" + "Add override" CTA from copy registry).
3. **Customer exists and has overrides** → table of overrides rendered.

The edit variant page applies the same first branch (only customer-not-found triggers NotFound there — an edit page with zero overrides still shows the empty form so the user can add the first override).
