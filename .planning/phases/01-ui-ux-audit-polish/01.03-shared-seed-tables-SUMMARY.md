## Barrel-import pattern for plan 07

Plan 07's per-entity seeds (`equipment.ts`, `repair.ts`, `survey.ts`, …) consume via a single import line:

```typescript
import {
  depots,
  customers,
  lessors,
  surveyors,
  bicOwnerCodes,
  isoSizeTypes,
} from '@/data/seed/_shared';
```

Types are re-exported alongside data (`Customer`, `CustomerTier`, `Depot`, `CountryCode`, `Lessor`, `Surveyor`, `BicOwnerEntry`, `BicOwnerRole`, `IsoSizeType`, `IsoCategory`) so consumers can type their own lookups without reaching into individual files.

## Cross-table FK chain (for plan 07 equipment seed)

```
equipment.containerNumber  →  4-letter BIC prefix
                          ↓
            bicOwnerCodes (lookup by code)
                          ↓
       ┌──────────────────┴──────────────────┐
       ↓ role='carrier'                      ↓ role='lessor'
  customerCode → customers.code        lessorCode → lessors.code
       ↓                                     ↓
  Customer.tier (for tier-based            Lessor.name (for ownership
   pricing in tariff plans)                 reporting)
```

## Decisions made

1. **`C-{BIC}` customer code convention** — makes the FK chain trivially derivable from the BIC prefix printed on the container; no explicit join table needed.
2. **`L-{4letter}` lessor code convention** — short, mnemonic; lessors carry `bicOwnerCodes: string[]` to model multi-code owners (Triton dry + reefer).
3. **Tier mix collapsed for 10-customer set** — original UI-SPEC §9.6 mix was 2P/3G/5S/4Standard/2Bronze across 16 customers; with only 10 shipping-line customers here we use 2P/3G/4S/1Standard. Full mix lands in Phase 2 when lessors become billable customers.
4. **No BIC check-digit import in `bic-owner-codes.ts`** — owner codes are 4 letters only; the check-digit util applies to the full 11-char container number, which is constructed in plan 07's `equipment.ts`. Keeps this registry import-free and avoids circular dependency between seed `_shared/` and `_shared/`-consuming entities.
5. **BULK + FLAT shipped as stubs** per REQUIREMENTS.md EQUIP-03 — selectable in pickers but no dedicated workflows in v1.
